## 1. Propósito del documento

Este documento define la visión general de **CourtFlow** como producto.

Su objetivo es responder:

> ¿Qué es CourtFlow, qué problema resuelve, para quién está diseñado y hacia dónde debe evolucionar?

Este documento no define todavía tecnologías, base de datos, APIs ni pantallas específicas. Es el documento base sobre el cual se construirán las especificaciones posteriores de arquitectura, módulos, permisos, modelo de datos y experiencia de usuario.

---

## 2. Definición del producto

**CourtFlow** será una plataforma SaaS para la gestión integral de clubes de pádel.

El sistema estará orientado inicialmente a clubes de pádel, con posibilidad futura de extenderse a otros deportes de raqueta, especialmente tenis.

La plataforma deberá permitir que cada club administre su operación diaria desde un único sistema web, accesible desde computadoras, tablets y teléfonos móviles.

CourtFlow deberá ser modular, escalable, configurable y preparado para operar con múltiples clubes dentro de una misma plataforma.

---

## 3. Visión a largo plazo

La visión a largo plazo es convertir a CourtFlow en la plataforma de gestión para clubes de pádel más integrada, abarcativa y reconocida de Argentina, con posibilidad de expansión futura a Latinoamérica y otros mercados.

A largo plazo, CourtFlow deberá ser reconocido por:

- Ser el producto más integrado del mercado.
- Cubrir la mayor cantidad de procesos operativos de un club.
- Mantener una experiencia simple para usuarios no técnicos.
- Ofrecer estabilidad y confianza operativa.
- Tener un precio razonable y competitivo.
- Brindar atención personalizada.
- Evolucionar constantemente mediante innovación funcional y tecnológica.

---

## 4. Problema que resuelve

Muchos clubes de pádel administran su operación diaria utilizando herramientas separadas, poco integradas o manuales.

Problemas habituales:

- Uso excesivo de WhatsApp para reservas, torneos, pagos y consultas.
- Planillas de Excel para administrar jugadores, pagos, torneos o turnos.
- Falta de integración entre turnos, caja, socios, torneos, ranking y bar.
- Mucho tiempo perdido en tareas repetitivas.
- Dificultad para saber quién pagó y quién no.
- Comunicación manual con jugadores.
- Falta de estadísticas confiables.
- Dificultad para controlar la operación diaria.
- Dependencia de una persona que concentra la información.
- Sistemas existentes con precios altos o poco adaptados a la realidad local.

CourtFlow deberá resolver estos problemas centralizando la información y automatizando procesos clave.

## 6. Cliente objetivo

El cliente inicial será el **dueño del club de pádel**.

Este perfil será quien normalmente:

- Decide contratar el sistema.
- Paga la suscripción mensual.
- Busca mejorar la administración del club.
- Necesita reducir tiempo operativo.
- Quiere controlar mejor ingresos, turnos, socios, torneos y servicios.

También podrán ser clientes futuros:

- Socios propietarios de clubes.
- Administradores de complejos deportivos.
- Cadenas o franquicias de clubes.
- Organizadores de torneos.
- Asociaciones o federaciones.
- Complejos deportivos con múltiples deportes.

La arquitectura deberá permitir esta evolución, aunque el foco inicial sea el club individual.

Usuarios previstos:

- Visitante no autenticado.
- Jugador.
- Participante de torneo.
- Socio.
- Profesor.
- Recepcionista.
- Empleado del bar.
- Cajero.
- Organizador de torneos.

## 8. Modelo SaaS

CourtFlow será una plataforma SaaS.

Esto significa que:

- No se instalará una copia del sistema por cada club.
- Todos los clubes utilizarán una única plataforma centralizada.
- Cada club tendrá sus datos aislados.
- Las actualizaciones serán aplicadas centralmente.
- El acceso será mediante navegador web.
- La contratación será mediante suscripción mensual.

Cada club podrá tener un plan contratado con módulos habilitados según su suscripción.

---

## 9. Registro self-service

La plataforma deberá permitir que un club pueda registrarse directamente desde la web.

El flujo esperado será:

1. El usuario ingresa al sitio.
2. Crea una cuenta.
3. Registra los datos del club.
4. Configura información básica.
5. Selecciona un plan o prueba inicial.
6. Comienza a utilizar el sistema.

También deberán poder registrarse jugadores, participantes y otros perfiles, siempre respetando los permisos correspondientes.

El modelo comercial será por suscripción mensual.

El precio podrá depender de:

- Tamaño del club.
- Cantidad de canchas.
- Cantidad de usuarios administrativos.
- Funcionalidades avanzadas.
- Integraciones.
- Personalizaciones.

Ejemplos de planes futuros:

## 11. Personalización

CourtFlow deberá permitir dos niveles de personalización.

### 11.1 Configuración estándar

Cada club podrá configurar aspectos como:

- Datos del club.
- Cantidad de canchas.
- Horarios de atención.
- Módulos habilitados.
- Categorías.
- Reglas de reservas.
- Reglas de torneos.
- Notificaciones.
- Logo.
- Colores.
- Usuarios.
- Permisos.

Esta personalización deberá realizarse sin modificar código.

### 11.2 Funcionalidades exclusivas

Algunos clubes podrán contratar funcionalidades específicas desarrolladas especialmente para ellos.

Estas funcionalidades deberán implementarse sin afectar al núcleo común de la plataforma ni comprometer el mantenimiento general del sistema.

En documentos posteriores se definirá si esto se resuelve mediante:

- Feature flags.
- Módulos opcionales.
- Extensiones.
- Configuraciones avanzadas.
- Desarrollo específico controlado.

## 12. Alcance deportivo inicial y futuro

El alcance inicial será pádel.

Sin embargo, la arquitectura deberá prever una posible extensión futura a tenis u otros deportes de raqueta.

Esto implica evitar decisiones rígidas que impidan evolucionar hacia una plataforma multi-deporte.

No obstante, durante las primeras fases del producto, todas las decisiones funcionales estarán optimizadas para pádel.

---

## 13. Alcance geográfico

El foco inicial será Argentina.

Sin embargo, la plataforma deberá estar preparada para operar en otros países.

Esto implica prever desde el inicio:

- Múltiples idiomas.
- Monedas configurables.
- Formatos de fecha configurables.
- Zonas horarias por club.
- Configuraciones fiscales futuras.
- Pasarelas de pago por país.
- Textos traducibles.
- Reglas regionales configurables.

No será necesario implementar toda la internacionalización en la primera versión, pero la arquitectura no deberá impedirla.

---

## 14. Principios irrenunciables

CourtFlow deberá proteger siempre los siguientes principios:

### 14.1 Simplicidad de uso

El sistema deberá ser fácil de entender para usuarios no técnicos.

Un club no debería necesitar capacitación extensa para comenzar a usarlo.

### 14.2 Estabilidad

La plataforma deberá ser confiable. Los clubes dependerán del sistema para su operación diaria, por lo que la estabilidad será un valor central.

### 14.3 Integración total

Los módulos no deberán sentirse como aplicaciones separadas. Turnos, torneos, caja, socios, bar, ranking, clases, videos y notificaciones deberán integrarse de forma coherente.

### 14.4 Innovación constante

El producto deberá evolucionar permanentemente, incorporando mejoras funcionales, automatizaciones y nuevas tecnologías cuando aporten valor real.

## 16. Qué NO debe ser CourtFlow

CourtFlow no deberá convertirse en:

- Un sistema difícil de usar.
- Una aplicación lenta.
- Un conjunto de módulos desconectados.
- Un software dependiente de una PC local.
- Un producto rígido que no pueda adaptarse.
- Una solución costosa para clubes pequeños.
- Un sistema que requiera conocimientos técnicos avanzados.
- Un desarrollo con versiones distintas e incompatibles por cliente.
- Una plataforma que priorice funciones complejas por encima de la simplicidad.

## 17. Primer módulo previsto

El primer módulo será:

> Gestión de Torneos e Inscripción Online.

Este módulo deberá permitir:

- Crear torneos.
- Generar enlaces públicos de inscripción.
- Registrar participantes.
- Cargar disponibilidad horaria.
- Administrar inscripciones.
- Confirmar pagos.
- Preparar datos para armado de parejas, zonas y fixture.

Aunque este será el primer módulo, deberá diseñarse como parte de una plataforma mayor.

## 19. Criterios de éxito

El producto será considerado exitoso si logra:

- Reducir el tiempo operativo de los clubes.
- Centralizar la gestión en una única plataforma.
- Ser fácil de usar para perfiles no técnicos.
- Permitir crecimiento modular.
- Tener buena adopción por parte de clubes pequeños y medianos.
- Mantener precio competitivo.
- Convertirse en una referencia del mercado argentino.
- Evolucionar sin rehacer su arquitectura base.
- Soportar múltiples clubes sin pérdida de orden ni estabilidad.

---

## 20. Riesgos principales

Riesgos identificados:

- Intentar abarcar demasiados módulos demasiado pronto.
- Agregar personalizaciones sin control arquitectónico.
- Perder simplicidad por exceso de funciones.
- No definir correctamente permisos y roles.
- No diseñar bien el modelo multi-club.
- Crear dependencias fuertes entre módulos.
- Subestimar la importancia de la experiencia de usuario.
- No documentar decisiones importantes.
- Permitir que Cursor implemente sin especificación clara.

Estos riesgos deberán ser considerados en documentos posteriores.

## 23. Resumen ejecutivo

CourtFlow será una plataforma SaaS integral para clubes de pádel, diseñada inicialmente para Argentina pero preparada para escalar a otros países y eventualmente a otros deportes de raqueta.

Su objetivo es centralizar la operación completa de un club en una única solución web, simple, estable, integrada, configurable y de precio razonable.

El producto deberá diferenciarse por su integración total, facilidad de uso, atención personalizada, automatización, innovación constante y conocimiento real de las necesidades de los clubes de pádel.

La visión final es que CourtFlow se convierta en la plataforma más reconocida de Argentina para la gestión de clubes de pádel.
